# P8.1H Container Deployment Topology

This set defines the first cloud deployment topology for the migration runtime without changing runtime code or package references.

## Purpose

P8.1H establishes container and Azure hosting guidance for:

- `Migration.Admin.Api` as the operational control-plane HTTP surface.
- `Migration.Hosts.SqlOperationalWorker` as the SQL-backed worker host.
- `Migration.Workers.ServiceBusDispatcher` as the Service Bus dispatch worker.
- `Migration.Workers.ServiceBusExecutor` as the Service Bus execution worker.

## Hosting model

### Recommended first Azure target

Use Azure Container Apps for workers and either Azure Container Apps or App Service for the Admin API.

Recommended split:

| Component | Hosting | Scale driver | Notes |
|---|---|---|---|
| Admin API | App Service or Container Apps | HTTP | Exposes `/health/live` and `/health/ready`. |
| SQL Operational Worker | Container Apps job/service | SQL/runnable-run polling | No `RunId` secret. Idle is healthy. |
| Service Bus Dispatcher | Container Apps | SQL + queue pressure | Dispatches work to Service Bus. |
| Service Bus Executor | Container Apps | Service Bus queue length | Executes distributed work. |

## Required environment variables

Use the existing `MIGRATION_` prefix and double underscores for hierarchy.

Required SQL worker settings:

```text
MIGRATION_ConnectionStrings__MigrationOperationalStore
MIGRATION_SqlOperationalQueueExecutor__Enabled=true
MIGRATION_SqlOperationalQueueExecutor__WorkerId=<cloud-worker-id>
MIGRATION_SqlOperationalQueueExecutor__BatchSize=10
MIGRATION_SqlOperationalQueueExecutor__LeaseSeconds=300
MIGRATION_SqlOperationalQueueExecutor__PollDelaySeconds=5
MIGRATION_SqlOperationalQueueExecutor__RunUntilIdleAndStop=false
MIGRATION_SqlOperationalMigrationJobExecutor__Enabled=false
```

Do not configure `MIGRATION_SqlOperationalQueueExecutor__RunId` except for short-lived debug runs.

## Probe contract

Admin API:

- `/health/live` for liveness.
- `/health/ready` for readiness.

Worker containers:

- Liveness is process/container liveness.
- Readiness is startup readiness and logs from `SqlOperationalWorkerStartupProbe`.
- Idle with no runnable runs is healthy.

## Dockerfile strategy

This set includes Dockerfile templates under `deploy/docker/`.

They are intentionally templates because repository build context, publish trimming, and image registry details should be selected during deployment automation.

## Out of scope

This set intentionally does not add:

- Bicep/Terraform.
- CI/CD workflow files.
- package references.
- project-file changes.
- worker runtime changes.
- SQL schema changes.
