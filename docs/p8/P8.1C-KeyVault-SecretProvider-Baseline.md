# P8.1C Key Vault and Secret Provider Baseline

## Purpose

P8.1C prepares the SQL operational worker for Azure-hosted secret management without changing runtime code or package references yet.

The repo already uses provider-based configuration and the `MIGRATION_` environment-variable prefix. This set documents the cloud secret layout, the exact local-vs-cloud configuration boundary, and the validation steps needed before adding any Key Vault package/provider code.

## Current cloud-safe configuration model

The SQL operational worker should continue to read configuration from:

1. `appsettings.json`
2. environment-specific appsettings files
3. local-only appsettings files
4. user secrets for local development
5. environment variables with the `MIGRATION_` prefix

For Azure hosting, secrets should be supplied through one of these provider-backed mechanisms:

- Azure App Service / Container Apps app settings that reference Key Vault secrets.
- Azure Key Vault references in app settings.
- A later explicit Azure Key Vault configuration provider, if package coverage is confirmed and central package management is updated intentionally.

## Required production secrets/settings

The worker requires these cloud configuration values:

```json
{
  "ConnectionStrings:MigrationOperationalStore": "<sql-connection-string-or-provider-reference>",
  "SqlOperationalQueueExecutor:Enabled": "true",
  "SqlOperationalQueueExecutor:WorkerId": "<stable-worker-id-or-instance-template>",
  "SqlOperationalQueueExecutor:BatchSize": "10",
  "SqlOperationalQueueExecutor:LeaseSeconds": "300",
  "SqlOperationalQueueExecutor:PollDelaySeconds": "5",
  "SqlOperationalQueueExecutor:RunUntilIdleAndStop": "false",
  "SqlOperationalMigrationJobExecutor:Enabled": "false"
}
```

Do not configure `SqlOperationalQueueExecutor:RunId` in production. It is a short-lived local/debug override only.

## Azure app setting names

Because the worker uses `AddEnvironmentVariables(prefix: "MIGRATION_")`, cloud app settings can be represented as double-underscore environment variable keys:

```text
MIGRATION_ConnectionStrings__MigrationOperationalStore
MIGRATION_SqlOperationalQueueExecutor__Enabled
MIGRATION_SqlOperationalQueueExecutor__WorkerId
MIGRATION_SqlOperationalQueueExecutor__BatchSize
MIGRATION_SqlOperationalQueueExecutor__LeaseSeconds
MIGRATION_SqlOperationalQueueExecutor__PollDelaySeconds
MIGRATION_SqlOperationalQueueExecutor__RunUntilIdleAndStop
MIGRATION_SqlOperationalMigrationJobExecutor__Enabled
```

## Key Vault reference candidates

Recommended Key Vault secret names:

```text
migration-operational-store-connection-string
sql-operational-queue-executor-worker-id
```

The connection string should be secret-backed. Most queue executor tuning values can remain plain app settings unless governance requires all configuration to come from Key Vault.

## Managed identity expectations

The cloud host should use a managed identity with least-privilege access to:

- read Key Vault secrets needed by the host;
- connect to Azure SQL if using Azure AD authentication;
- write telemetry to the configured monitoring destination.

SQL permissions should be constrained to the operational schema used by the worker.

## P8.1D handoff

After this baseline is validated, the next safe implementation step is health/readiness endpoint and cloud probe alignment, followed by explicit Key Vault provider code only if environment-provider Key Vault references are insufficient.
