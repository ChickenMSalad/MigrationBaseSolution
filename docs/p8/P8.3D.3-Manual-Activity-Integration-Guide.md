# P8.3D.3 Manual Activity Integration Guide

This set intentionally contains **no repair scripts** and **no mutation scripts**.

Use it to manually wire OpenTelemetry `Activity` usage into the three runtime boundaries already identified by the P8.3D.1/P8.3D.2 inventories:

- SQL queue execution
- Service Bus execution
- Service Bus dispatch

The goal is to preserve existing behavior while adding Activity spans around existing execution/send boundaries.

## Preconditions

These files should already exist from P8.3B/P8.3C:

- `src/Core/Migration.Application/Operational/Telemetry/OperationalExecutionActivity.cs`
- `src/Core/Migration.Application/Operational/Telemetry/OperationalExecutionActivitySources.cs`
- `src/Core/Migration.Application/Operational/Telemetry/OperationalExecutionActivityTags.cs`
- `src/Core/Migration.Application/Operational/Telemetry/OperationalExecutionTelemetryFields.cs`

Do not proceed if `dotnet build` is currently failing.

## 1. SQL queue worker Activity usage

File:

```text
src/Workers/Migration.Workers.QueueExecutor/Services/SqlOperationalWorkItemWorker.cs
```

Add using if missing:

```csharp
using Migration.Application.Operational.Telemetry;
```

Inside `ExecuteClaimedItemAsync`, after the existing telemetry `BeginScope(...)` block and before:

```csharp
var result = await _executor.ExecuteAsync(item, cancellationToken).ConfigureAwait(false);
```

add:

```csharp
using var activity = OperationalExecutionActivity.StartSqlQueueWorkItemExecution(
    item.RunId,
    item.WorkItemId,
    item.ManifestRowId,
    item.WorkItemType,
    item.AttemptCount,
    item.PartitionKey);
```

After the executor call returns, before completion/failure handling branches, add:

```csharp
OperationalExecutionActivity.SetExecutionDuration(activity, DateTimeOffset.UtcNow - startedAtUtc);
OperationalExecutionActivity.SetExecutionResult(activity, result.Succeeded, result.ErrorCode);
```

If your result variable is scoped inside a branch, place the result tagging immediately after the existing executor call and before the first `if (result.Succeeded)` branch.

Preserve all existing calls to:

- `CompleteAsync(...)`
- `FailAsync(...)`
- execution-history recording
- retry/next-attempt behavior

## 2. Service Bus executor Activity usage

File:

```text
src/Workers/Migration.Workers.ServiceBusExecutor/Runtime/SqlServiceBusExecutorWorker.cs
```

Add using if missing:

```csharp
using Migration.Application.Operational.Telemetry;
```

Find the existing telemetry scope around the resolved `workItem`. It should be near:

```csharp
using var telemetryScope = _logger.BeginScope(new Dictionary<string, object?>
```

Immediately after that scope and before:

```csharp
var result = await _executor.ExecuteAsync(workItem, args.CancellationToken);
```

add:

```csharp
var activityStartedAtUtc = DateTimeOffset.UtcNow;
using var activity = OperationalExecutionActivity.StartServiceBusWorkItemExecution(
    workItem.RunId,
    workItem.WorkItemId,
    workItem.ManifestRowId,
    workItem.WorkItemType,
    workItem.AttemptCount,
    workItem.PartitionKey,
    args.Message.CorrelationId,
    args.Message.MessageId,
    args.Message.DeliveryCount);
```

After the executor call returns and before queue settlement, add:

```csharp
OperationalExecutionActivity.SetExecutionDuration(activity, DateTimeOffset.UtcNow - activityStartedAtUtc);
OperationalExecutionActivity.SetExecutionResult(activity, result.Succeeded, result.ErrorCode);
```

Preserve all existing settlement behavior:

- invalid messages still dead-letter
- missing work items still dead-letter
- successful SQL completion still completes the Service Bus message
- retryable failures still complete SQL failure and settle according to existing logic
- terminal failures still dead-letter according to existing logic
- exceptions still abandon according to existing logic

## 3. Service Bus dispatcher Activity usage

File:

```text
src/Workers/Migration.Workers.ServiceBusDispatcher/Dispatching/SqlWorkItemDispatcher.cs
```

Add using if missing:

```csharp
using Migration.Application.Operational.Telemetry;
```

Find the message creation/send boundary:

```csharp
ServiceBusMessage message = new(json)
```

and:

```csharp
await _sender!.SendMessageAsync(message, cancellationToken).ConfigureAwait(false);
await MarkDispatchedAsync(item.WorkItemId, cancellationToken).ConfigureAwait(false);
```

Immediately before `SendMessageAsync(...)`, add:

```csharp
var activityStartedAtUtc = DateTimeOffset.UtcNow;
using var activity = OperationalExecutionActivity.StartServiceBusDispatch(
    item.RunId,
    item.WorkItemId,
    item.ManifestRowId,
    item.WorkItemType,
    item.AttemptCount,
    item.PartitionKey,
    message.CorrelationId,
    message.MessageId);
```

Immediately after `MarkDispatchedAsync(...)`, add:

```csharp
OperationalExecutionActivity.SetExecutionDuration(activity, DateTimeOffset.UtcNow - activityStartedAtUtc);
OperationalExecutionActivity.SetExecutionResult(activity, succeeded: true, errorCode: null);
```

Do not change the serialized message contract unless a compile error proves it is required.

## Validation

After manual edits, run:

```powershell
dotnet build
```

Then inspect the changed files:

```powershell
git diff -- src/Workers/Migration.Workers.QueueExecutor/Services/SqlOperationalWorkItemWorker.cs
git diff -- src/Workers/Migration.Workers.ServiceBusExecutor/Runtime/SqlServiceBusExecutorWorker.cs
git diff -- src/Workers/Migration.Workers.ServiceBusDispatcher/Dispatching/SqlWorkItemDispatcher.cs
```

The diff should show only Activity instrumentation and using statements.

## What this completes

After this patch, runtime observability should include:

- structured logging scopes
- canonical telemetry fields
- ActivitySource spans
- SQL queue execution activities
- Service Bus execution activities
- Service Bus dispatch activities
- duration/result tagging

Exporter registration and Azure Monitor wiring should be handled separately in a later P8.3 set.
