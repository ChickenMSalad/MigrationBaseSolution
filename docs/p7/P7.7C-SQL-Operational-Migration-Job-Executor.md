# P7.7C SQL Operational Migration Job Executor

This set replaces the P7.7B safe logging/no-op work-item executor with an opt-in `IMigrationJobRunner` backed executor for SQL operational work items.

## Payload contract

Each SQL operational work item payload must contain either a `MigrationJobDefinition` JSON object at the root or one of these wrapper properties:

```json
{
  "job": {
    "jobName": "example",
    "sourceType": "LocalStorage",
    "targetType": "LocalStorage",
    "manifestType": "Csv",
    "manifestPath": "...",
    "settings": {},
    "dryRun": true,
    "parallelism": 4
  }
}
```

Accepted wrapper names:

- `job`
- `migrationJob`
- `jobDefinition`
- `migrationJobDefinition`

Required job fields:

- `jobName`
- `sourceType`
- `targetType`
- `manifestType`

## Configuration

The executor is opt-in:

```json
{
  "SqlOperationalMigrationJobExecutor": {
    "Enabled": false,
    "AddOperationalContextSettings": true,
    "TreatJobFailuresAsRetryable": false,
    "TreatUnhandledExceptionsAsRetryable": true,
    "RetryDelaySeconds": 300
  }
}
```

Set `SqlOperationalMigrationJobExecutor:Enabled=true` only after the P7.7B SQL lifecycle smoke path has been validated.

## Operational behavior

When enabled, the SQL operational worker flow is:

1. Claim SQL work item through `IOperationalWorkItemQueue`.
2. Deserialize the work-item payload into `MigrationJobDefinition`.
3. Add SQL operational context into the job settings bag.
4. Hydrate project credentials through `ProjectCredentialJobSettingsHydrator`.
5. Execute the existing orchestration path through `IMigrationJobRunner`.
6. Complete the SQL work item if the job summary has zero failures.
7. Fail or retry the SQL work item based on executor options when job execution fails.

## Apply

From repository root:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P7SqlOperationalMigrationJobExecutor.ps1
dotnet build
```

## Important

This set intentionally executes whole migration jobs from SQL work-item payloads. It does not yet implement one-manifest-row-per-work-item connector execution. That row-level executor should be the next set after this proves that SQL claim/execute/complete can call the existing orchestration runtime successfully.
