# P10.2Z - Admin Web Feature Folder Structure

## Purpose

P10.2Z locks the canonical Admin Web feature-folder structure after the P10.2 consolidation work.

The canonical deployable UI remains:

`src/Admin/Migration.Admin.Web`

The feature-source/reference UI remains:

`apps/migration-admin-ui`

This set does not move or delete feature files. It adds a canonical feature grouping guide under `src/Admin/Migration.Admin.Web/src/features` and a report script that inventories the current canonical pages, API clients, types, components, and remaining feature-source folders.

## Canonical grouping model

Future Admin Web work should use this grouping model:

| Canonical group | Intended contents |
| --- | --- |
| `features/operations` | Runtime dashboard, run detail, execution sessions, failure retry, worker telemetry, operational events |
| `features/connectors` | Connector catalog, connector configuration, execution profiles |
| `features/security` | Credentials and credential vault surfaces |
| `features/governance` | Audit trail, notification routing, command center, capacity forecast, cost analytics |
| `features/platform` | Cross-cutting shell, navigation, shared UI composition guidance |

The existing top-level `pages`, `api`, and `types` folders remain supported during consolidation. This set creates a structure guide so later sets can move code in small, build-safe batches rather than flattening everything forever.

## Rules

- Do not add new Admin UI feature work under `apps/migration-admin-ui`.
- Do not delete `apps/migration-admin-ui` until the remaining useful feature-source components have been migrated or explicitly rejected.
- Do not perform broad automated source rewrites against `App.tsx` or `Layout.tsx` unless the current repo shape is verified first.
- Prefer small feature-family moves with build validation after each set.
