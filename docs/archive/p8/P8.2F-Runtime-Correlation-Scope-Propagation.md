# P8.2F Runtime Correlation Scope Propagation

This set adds a small reusable telemetry scope helper and guarded repair scripts for applying `ILogger.BeginScope(...)` around SQL and Service Bus work-item execution paths.

## Goals

- Keep `RunId` and `WorkItemId` present on runtime logs.
- Preserve existing SQL operational execution history behavior.
- Preserve existing Service Bus settlement behavior.
- Avoid introducing OpenTelemetry package dependencies in this set.
- Avoid broad source mutation.

## Scope helper

`OperationalExecutionTelemetryScope.Create(...)` returns a dictionary suitable for:

```csharp
using var scope = _logger.BeginScope(OperationalExecutionTelemetryScope.Create(...));
```

The helper uses the existing `OperationalExecutionTelemetryFields` constants for known canonical fields and local literals for worker/Service Bus transport fields.

## Guarded repair scripts

The repair scripts only patch files when the expected source patterns are present and the scope has not already been added. If a source file has drifted, the script stops instead of guessing.

## Runtime policy

- SQL queue worker scopes should include `RunId`, `WorkItemId`, and `WorkerId`.
- Service Bus executor scopes should include `RunId`, `WorkItemId`, `WorkerId`, `ServiceBusMessageId`, `ServiceBusCorrelationId`, and delivery count when available.
- Debug-only `RunId` configuration remains prohibited for production cloud worker templates.
