# P8.2F.1 Service Bus Runtime Correlation Completion

## Purpose

P8.2F introduced canonical operational execution telemetry fields and SQL queue worker scope usage. The remaining gap is the Service Bus executor runtime path: Service Bus message processing should emit the same `RunId` / `WorkItemId` / Service Bus correlation scope before invoking the real SQL operational execution adapter.

This set is intentionally small and corrective:

- complete Service Bus executor `ILogger.BeginScope(...)` propagation
- use existing `OperationalExecutionTelemetryScope`
- preserve existing completion, failure, abandon, and dead-letter behavior
- avoid SQL, package, project, and schema changes

## Expected runtime behavior

For each Service Bus-delivered work item, logs emitted while executing the operational work item should be enriched with the canonical operational correlation values:

- `RunId`
- `WorkItemId`
- optional `ServiceBusCorrelationId`

The Service Bus executor should continue to:

- dead-letter invalid messages
- dead-letter missing SQL work items
- complete successful work items
- fail or abandon unsuccessful work items according to existing logic

## Apply

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Repair-P82F1ApplyServiceBusRuntimeCorrelationScope.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P82F1ServiceBusRuntimeCorrelationCompletion.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\Write-P82F1ServiceBusRuntimeCorrelationInventory.ps1
dotnet build
```

## Notes

This set does not change the Service Bus message contract and does not introduce a parallel execution path. It only adds runtime correlation scope around the existing Service Bus executor invocation boundary.
