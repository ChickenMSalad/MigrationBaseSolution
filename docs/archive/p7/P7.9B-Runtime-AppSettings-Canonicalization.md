# P7.9B Runtime AppSettings Canonicalization

## Purpose

P7.9B removes runtime ambiguity caused by duplicate App Service configuration keys. It does not delete settings automatically. It inventories current settings, validates the canonical contract, and generates reviewable Azure CLI commands for applying canonical keys and deleting known stale keys.

## Canonical runtime keys

Dispatcher host:

- `ConnectionStrings__MigrationOperationalStore`
- `SqlServiceBusDispatcher__Enabled`
- `SqlServiceBusDispatcher__SqlConnectionString`
- `SqlServiceBusDispatcher__ServiceBusConnectionString`
- `SqlServiceBusDispatcher__QueueName`
- `OpenTelemetry__EnableTracing`
- `OpenTelemetry__EnableAzureMonitorExporter`

Executor host:

- `ConnectionStrings__MigrationOperationalStore`
- `SqlServiceBusExecutor__ServiceBusConnectionString`
- `SqlServiceBusExecutor__QueueName`
- `SqlOperationalWorkItemQueue__SchemaName`
- `SqlOperationalWorkItemQueue__WorkItemsTableName`
- `SqlOperationalMigrationJobExecutor__Enabled`
- `GenericMigrationRuntime__RegisterAllWhenEmpty`
- `OpenTelemetry__EnableTracing`
- `OpenTelemetry__EnableAzureMonitorExporter`

## Known stale or transitional families

These should not be used by the runtime hosts after cleanup:

- `MIGRATION_ConnectionStrings__MigrationOperationStore`
- `MIGRATION_SqlWorkItemDispatcher__*`
- `MIGRATION_ServiceBusDispatcher__*`
- `MIGRATION_SqlServiceBusExecutor__*`
- `MIGRATION_SqlOperationalWorkItemQueue__*`
- `MIGRATION_SqlOperationalMigrationJobExecutor__*`
- `OperationalStore__Sql__*`

The cleanup scripts generate delete commands for these families, but the operator must review and run those commands explicitly.

## Recommended flow

1. Export live settings from Azure.
2. Run `Get-RuntimeCanonicalAppSettingsPlan.ps1` against both exports.
3. Run `Test-RuntimeCanonicalAppSettings.ps1` without `-FailOnLegacy` to identify missing canonical keys.
4. Generate cleanup commands with `New-RuntimeCanonicalAppSettingsApplyCommands.ps1`.
5. Review the generated commands.
6. Apply canonical setting commands first.
7. Restart workers.
8. Confirm runtime starts.
9. Apply stale-key delete commands.
10. Re-export settings and rerun validation with `-FailOnLegacy`.

## Notes

This set intentionally avoids automatic deletion because the dev cloud currently contains manually patched settings from live recovery work. The objective is a controlled transition to a single settings surface, not another patch spiral.
