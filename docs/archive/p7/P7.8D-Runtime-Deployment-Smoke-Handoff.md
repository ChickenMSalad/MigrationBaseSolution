# P7.8D Runtime Deployment Smoke Handoff

## Purpose

This set turns the Service Bus dispatcher/executor cloud smoke flow into a repeatable handoff workflow.

It intentionally does not mutate runtime code. It adds scripts and samples that make deployment and smoke validation explicit so that future failures are caught before live debugging begins.

## Background

The P7 cloud smoke exposed several classes of issues:

- stale publish artifacts deployed to App Service
- mixed dispatcher/executor appsetting names
- old GUID-backed SQL tables coexisting with new bigint runtime tables
- worker SQL code accidentally resolving `dbo.MigrationWorkItems`
- queue payload contracts disagreeing about `WorkItemId` type
- executor runtime features disabled by configuration
- smoke payloads that were syntactically valid JSON but not valid `MigrationJobDefinition` payloads

P7.8A-C added runtime contract, SQL contract, and appsettings cleanup guardrails. P7.8D adds the deployment and smoke handoff workflow that must be used before calling the runtime deployable.

## Canonical deployment order

1. Confirm repo is clean.
2. Validate runtime contract scripts from P7.8A-C.
3. Publish/deploy dispatcher from clean output.
4. Publish/deploy executor from clean output.
5. Restart executor first.
6. Enqueue a fresh smoke work item.
7. Restart dispatcher.
8. Watch executor log and verify SQL final state.
9. Commit any changes before redeploying again.

## Canonical runtime smoke status

A handoff smoke test must show the item moving through this path:

```text
Queued -> Dispatched -> Completed
```

A controlled business failure can be accepted only when the failure message is expected and documented. Infrastructure failures are not acceptable handoff outcomes.

## Smoke payload

The sample smoke payload is a minimal `MigrationJobDefinition` shape:

```json
{
  "jobName": "RuntimeSmoke",
  "sourceType": "LocalStorage",
  "targetType": "LocalStorage",
  "manifestType": "Csv",
  "mappingProfilePath": "smoke.json",
  "dryRun": true
}
```

The specific provider values must match what the deployed executor registers. If `Csv`, `LocalStorage`, or another type is not registered, the smoke failure is a runtime composition failure, not a Service Bus or SQL failure.

## Acceptance criteria

- Worker publish output is deleted before each publish.
- Deployment ZIP is created only after publish succeeds.
- Smoke SQL uses `migration.WorkItems` with bigint identity `WorkItemId`.
- Smoke enqueue does not use inline PowerShell JSON quoting.
- Smoke verifier reports latest work items and Service Bus queue counts.
- Scripts are PowerShell 5.1 compatible and avoid destructive cleanup unless explicitly requested.
