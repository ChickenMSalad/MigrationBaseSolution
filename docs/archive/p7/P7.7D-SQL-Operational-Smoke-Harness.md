# P7.7D SQL Operational Smoke Harness

This set adds safe operational smoke-test helpers for the SQL operational worker path.

It does not add runtime code, project references, packages, or new namespaces.

## Purpose

P7.7B added SQL work-item polling and lifecycle behavior. P7.7C added an opt-in migration-job executor. Before enabling real migration execution, this set helps validate:

- SQL operational store readiness
- worker configuration
- safe no-op lifecycle behavior
- work item claim/complete/fail paths
- run summary visibility

## Safe default posture

Keep real migration execution disabled until the SQL lifecycle smoke path passes:

```json
"SqlOperationalMigrationJobExecutor": {
  "Enabled": false
}
```

Use no-op completion only for lifecycle validation:

```json
"SqlOperationalWorker": {
  "Enabled": true,
  "CompleteNoOpWorkItems": true
}
```

After lifecycle validation, set `CompleteNoOpWorkItems` back to false before enabling the migration-job executor.

## Apply

Copy this set into the repository root.

Run:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P7SqlOperationalSmokeHarness.ps1
```

## Configure local user secrets

From the repo root, run:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Set-P7SqlOperationalWorkerLocalSecrets.ps1 `
  -ConnectionString "Server=localhost;Database=MigrationOperationalStore;Trusted_Connection=True;TrustServerCertificate=True" `
  -EnableWorker $true `
  -CompleteNoOpWorkItems $true `
  -EnableMigrationJobExecutor $false
```

## Optional SQL smoke data

Review before executing:

```text
database\sql\p7\004_smoke_seed_operational_run.sql
```

This script is intentionally conservative and idempotent-ish around a fixed smoke `RunKey`.

## Run host

```powershell
dotnet run --project .\src\Hosts\Migration.Hosts.SqlOperationalWorker\Migration.Hosts.SqlOperationalWorker.csproj --environment Development
```

## Expected result

With `CompleteNoOpWorkItems=true` and `SqlOperationalMigrationJobExecutor:Enabled=false`, the worker should be able to claim SQL work items and complete them through the safe no-op path.

## Before real execution

Set:

```json
"SqlOperationalWorker": {
  "CompleteNoOpWorkItems": false
},
"SqlOperationalMigrationJobExecutor": {
  "Enabled": true
}
```

Then run with a real `MigrationJobDefinition` payload only after connector credentials and manifest paths are confirmed.
