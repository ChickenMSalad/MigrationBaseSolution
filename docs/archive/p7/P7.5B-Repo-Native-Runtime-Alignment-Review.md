# P7.5B Repo-Native Runtime Alignment Review

This set is intentionally conservative. It does not add new runtime abstractions, does not mutate project files, and does not add packages.

## Purpose

P7.5A exposed that the repo already has an established runtime layout:

- `src/Core/Migration.Infrastructure`
- `src/Core/Migration.Infrastructure.Sql`
- `src/Core/Migration.GenericRuntime`
- `src/Core/Migration.Orchestration`
- `src/Workers/Migration.Workers.QueueExecutor`
- `src/Hosts/Migration.Hosts.SqlOperationalWorker`

P7.5B verifies that the repo is aligned before further implementation wiring continues.

## What this validates

- No invalid fabricated top-level runtime folders remain.
- No references to the deleted `Migration.Infrastructure.Runtime` namespace remain.
- P7 SQL scripts 001, 002, and 003 exist.
- SQL operational worker host exists under `src/Hosts`.
- Existing QueueExecutor, GenericRuntime, Orchestration, and Infrastructure.Sql projects exist.
- Project files do not contain inline `PackageReference Version` attributes.
- `bin` and `obj` are ignored using non-regex path checks.
- The solution does not reference invalid top-level `src/Migration.Worker` or `src/Migration.Infrastructure` projects.

## After drop-in

Run from repository root:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P7RepoNativeRuntimeAlignment.ps1
```

Optional inventory report:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Write-P7RuntimeInventory.ps1
```

The inventory report writes:

```text
artifacts/p7/P7-Runtime-Inventory.txt
```

## Important

This set intentionally does not create new host/runtime code. It is the checkpoint between the repaired P7.5A worker host and the next implementation set.

The next implementation set should wire into the existing repo-native runtime areas instead of reintroducing `Migration.Infrastructure.Runtime`.
