# P7.10 Cloud Operational Readiness

P7.10 is a closeout checkpoint for the SQL-backed operational runtime. It does not introduce broad runtime mutation. It validates that the repo has the correct worker host, queue executor, SQL runtime bootstrap, run discovery, and execution-history surfaces before moving into cloud deployment and real connector execution.

## What this set adds

- `scripts/Validate-P710CloudOperationalReadiness.ps1`
- `scripts/Write-P710CloudOperationalReadinessInventory.ps1`
- `docs/p7/P7.10-Cloud-Operational-Readiness.md`

## Validation

Run from the repository root:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P710CloudOperationalReadiness.ps1
```

Then generate the readiness inventory:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Write-P710CloudOperationalReadinessInventory.ps1
```

This writes:

```text
docs\p7\P7.10-Cloud-Operational-Readiness-Inventory.generated.md
```

## Local worker secrets baseline

Use only the operational queue executor settings for the SQL worker host:

```json
{
  "ConnectionStrings:MigrationOperationalStore": "Server=(localdb)\\MSSQLLocalDB;Database=MigrationOperationalStore;Trusted_Connection=True;TrustServerCertificate=True",
  "SqlOperationalQueueExecutor:Enabled": "true",
  "SqlOperationalQueueExecutor:WorkerId": "local-sql-operational-worker-01",
  "SqlOperationalQueueExecutor:BatchSize": "10",
  "SqlOperationalQueueExecutor:LeaseSeconds": "300",
  "SqlOperationalQueueExecutor:PollDelaySeconds": "5",
  "SqlOperationalQueueExecutor:RunUntilIdleAndStop": "false",
  "SqlOperationalMigrationJobExecutor:Enabled": "false"
}
```

Do not configure `SqlOperationalQueueExecutor:RunId` for normal local or cloud operation. It is only a short-lived debug override.

## Expected local idle behavior

With no runnable runs:

```text
SQL operational QueueExecutor worker starting. ... RunIdOverride=(null)
SQL operational worker host readiness check passed. Status=Ready
SQL operational queue idle. No runnable migration runs found.
```

## P7 exit posture

P7 can be considered operationally closed when:

- SQL bootstrap compatibility applies cleanly.
- Execution history schema validates.
- SQL worker host starts with no RunId secret.
- Readiness reports Ready.
- Worker discovers runnable runs from SQL.
- Empty queue produces stable idle behavior.
- No top-level `src\Migration.Infrastructure` or `src\Migration.Worker` paths exist.

The next phase should be cloud deployment and real execution activation, not more local SQL scaffolding.
