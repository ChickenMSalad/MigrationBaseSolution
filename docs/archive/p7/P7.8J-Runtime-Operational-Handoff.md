# P7.8J Runtime Operational Handoff

## Purpose

This set turns the P7.8 cleanup work into a handoffable operational package. It does not introduce runtime code changes. It documents the required deployment evidence, the expected operator sequence, and the rollback/triage decision points that must be satisfied before anyone treats the SQL + Service Bus runtime as deployable.

## Why this set exists

The runtime deployment exposed several classes of drift:

- SQL and code disagreed on `WorkItemId` / `ManifestRowId` types.
- Azure App Service settings had duplicated prefixed/unprefixed keys and stale names.
- Old GUID-era SQL objects existed beside the new bigint runtime objects.
- Build success was treated as deploy readiness even though schema/config/runtime contracts were not proven.
- Live debugging created state that was not handoffable without cleanup evidence.

This handoff set makes those risks explicit and gives operators a single checklist to follow.

## Canonical runtime contract

| Concern | Canonical value |
|---|---|
| Work item table | `migration.WorkItems` |
| Manifest row table | `migration.ManifestRows` |
| Work item id type | `bigint` / C# `long` |
| Manifest row id type | `bigint` / C# `long?` where nullable |
| Run id type | `uniqueidentifier` / C# `Guid` |
| Dispatcher host | `Migration.Workers.ServiceBusDispatcher` |
| Executor host | `Migration.Workers.ServiceBusExecutor` |
| Queue | `migration-operational-work-items` |

## Required handoff evidence

Before handoff, attach or store:

1. Git commit SHA used for deployment.
2. Clean `git status --short` output.
3. Runtime SQL contract validator output.
4. Runtime appsettings cleanup plan output.
5. Runtime deployment smoke output.
6. Queue count before and after smoke.
7. Work item final SQL state for the smoke work item.
8. Any known dead-letter count and whether it is historical or active.

## No-go conditions

Do not hand off as deploy-ready if any of these are true:

- The deployment came from a dirty working tree.
- App settings include both active canonical keys and stale duplicate keys that can bind to the same option object.
- The runtime worker code references `dbo.MigrationWorkItems` or `migration.MigrationWorkItems` for active queue execution.
- The SQL runtime contract validator fails.
- The smoke run fails before reaching `SqlOperationalMigrationJobWorkItemExecutor`.
- The Service Bus queue accumulates active messages without an understood executor outcome.
- The executor dead-letters new messages during the smoke run.

## Recommended sequence

1. Validate repo cleanup sets.
2. Validate runtime code contract.
3. Validate SQL contract against target database.
4. Export and review target App Service settings.
5. Apply only canonical App Service settings.
6. Publish dispatcher and executor from a clean commit.
7. Restart executor first.
8. Enqueue smoke work item.
9. Restart dispatcher.
10. Watch dispatcher logs for dispatch count.
11. Watch executor logs for work item execution.
12. Verify final SQL state.
13. Record handoff evidence.

## Rollback principle

Rollback must revert both code and configuration together. A previous binary with new settings, or a new binary with old settings, is not a rollback. The rollback record must identify:

- code commit
- deployed artifacts
- App Service setting snapshot
- SQL baseline state
- queue/dead-letter state
