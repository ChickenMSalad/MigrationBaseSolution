# P7.9E - Dev Cloud Reset Readiness

## Purpose

This set prepares a safe, repeatable reset path for the development cloud runtime after the deployment cleanup work exposed schema, settings, queue, and smoke-harness drift.

It does **not** wipe anything automatically.

The goal is to make a destructive reset an explicit operator decision after evidence has been exported and reviewed.

## Why this set exists

The runtime has now proven the platform path can work, but the dev cloud environment still has accumulated state from earlier drift:

- legacy GUID-era runtime objects still exist next to bigint runtime objects
- `migration.WorkItems` previously had FK parentage drift
- Service Bus has old dead-lettered smoke messages
- app settings previously had duplicate prefixed/unprefixed keys
- smoke data from failed attempts remains in SQL

A clean dev cloud reset may be the fastest way to get back to a trustworthy baseline, but it must be done from scripts and evidence, not portal guessing.

## Recommended sequence

1. Export current settings and schema using the P7.8 tooling.
2. Run the readiness SQL.
3. Generate the reset plan markdown.
4. Review the plan.
5. Only then decide whether to run the guarded cleanup template with `@AllowDestructiveReset = 1`.

## Non-goals

This set does not:

- drop databases
- delete Azure resources
- purge Service Bus messages automatically
- mutate app settings
- rewrite runtime code

Those should be deliberate follow-up steps after the readiness report is reviewed.
