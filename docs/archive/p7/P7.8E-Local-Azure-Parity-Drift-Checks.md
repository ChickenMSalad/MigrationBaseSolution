# P7.8E - Local/Azure Parity Drift Checks

## Purpose

This set adds repo-native tooling to compare runtime settings and SQL contract snapshots between local and Azure environments before a worker deployment is considered handoffable.

The previous cloud smoke revealed several drift classes that must now be detected before deployment:

- duplicate app setting families where one key is active and another is stale;
- old GUID-era operational tables coexisting with the bigint runtime queue tables;
- worker defaults pointing at legacy table names;
- Azure/local SQL schema divergence;
- deployment attempts from a repo state that had not been validated against live runtime configuration.

## Scope

This set is guardrail-only. It does not mutate runtime source code, SQL schema, or Azure app settings.

## Expected workflow

1. Export local or known-good runtime snapshots.
2. Export Azure runtime snapshots.
3. Compare settings/schema snapshots before deployment.
4. Generate a parity report.
5. Fail the handoff if critical drift is detected.

## Critical runtime invariants

The active SQL Service Bus runtime must use:

| Surface | Canonical value |
|---|---|
| Work item table | `migration.WorkItems` |
| Manifest row table | `migration.ManifestRows` |
| Work item id type | `bigint` / `long` |
| Manifest row id type | `bigint` / `long?` |
| Run id type | `uniqueidentifier` / `Guid` |
| Worker queue | `SqlServiceBusDispatcher` + `SqlServiceBusExecutor` |

The old GUID-era operational objects are allowed to exist during cleanup, but worker/runtime hosts must not depend on them.

## Handoff rule

Do not deploy dispatcher/executor workers unless:

- `Test-RuntimeSqlContract.ps1` passes against the target database;
- `Test-RuntimeAppSettings.ps1` passes against the target app settings export;
- `Test-RuntimeRepoCloudParity.ps1` reports no critical drift;
- the deployment is from a clean committed repo state.
