# P7.9C Execution History Wiring Readiness

## Purpose

P7.9A added the SQL execution-history schema and P7.9B added the execution-history writer service. P7.9C intentionally does not alter the worker execution path yet. It verifies the repo-native locations and generates a wiring inventory so the next set can safely connect execution attempts to the real queue executor without repeating the earlier path/schema mistakes.

## Why this set is conservative

The current runtime path is now operational:

- SQL worker host starts without a pinned `RunId` secret.
- Readiness checks pass.
- Runnable run discovery works.
- Queue/coordinator lifecycle works.
- The worker can reach a correct idle state.

The next behavior change will instrument actual work-item execution. That patch needs to target the exact implementation currently used by the repo, normally under:

- `src\Workers\Migration.Workers.QueueExecutor`
- `src\Core\Migration.Infrastructure.Sql\Operational\ExecutionHistory`
- `src\Core\Migration.Application\Operational\WorkItems`

P7.9C generates a local inventory of those concrete files and validates the execution-history writer baseline first.

## After drop-in

Run:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P79CExecutionHistoryWiringReadiness.ps1
powershell -ExecutionPolicy Bypass -File .\scripts\Write-P79CExecutionHistoryWiringInventory.ps1
```

The inventory is written to:

```text
docs\p7\P7.9C-Execution-History-Wiring-Inventory.generated.md
```

Commit the generated inventory if you want to preserve the current repo state before the next runtime behavior patch.

## Expected next set

P7.9D should wire execution-history recording around `ISqlOperationalWorkItemExecutor` / `SqlOperationalMigrationJobWorkItemExecutor` so every claimed work item creates an attempt record with:

- run id
- work item id
- manifest row id
- worker id
- work item type
- started/completed timestamps
- result status
- error code/message when applicable
- retryable flag
- payload/result JSON snippets where appropriate

The next set should remain additive where possible and avoid package changes.
