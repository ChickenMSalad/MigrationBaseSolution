# P7.7B SQL Operational Queue Adapter

This set wires the existing QueueExecutor worker surface to the existing SQL operational work-item queue.

It is repo-native and intentionally avoids the earlier mistakes:

- no `src/Migration.Infrastructure`
- no `src/Migration.Worker`
- no `Migration.Infrastructure.Runtime.*`
- no inline `PackageReference Version` attributes
- no parallel runtime framework

## Operational behavior

The SQL operational worker host now registers:

- SQL operational readiness checks
- SQL operational run coordinator
- SQL operational work item queue
- QueueExecutor SQL operational worker

The worker can:

1. optionally fan out a run using `IOperationalRunCoordinator.StartRunAsync`
2. claim SQL work items using `IOperationalWorkItemQueue.ClaimAsync`
3. execute each claimed item through `ISqlOperationalWorkItemExecutor`
4. complete successful items
5. fail retryable/non-retryable items
6. evaluate run completion after each batch

The default executor is intentionally conservative. It logs payload details and only completes work items when `SqlOperationalQueueExecutor:CompleteNoOpWorkItems` is true. This lets you validate SQL queue claim/complete/fail plumbing without accidentally performing real connector writes.

The next set should replace or decorate `ISqlOperationalWorkItemExecutor` with a real manifest-row/connector execution adapter once the exact manifest-row payload contract is confirmed.

## Apply

Copy the files into the repository root.

Run:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\Validate-P7SqlOperationalQueueAdapter.ps1
```

Then build:

```powershell
dotnet build
```

## Configure local secrets

From the host project folder:

```powershell
cd .\src\Hosts\Migration.Hosts.SqlOperationalWorker

dotnet user-secrets set "ConnectionStrings:MigrationOperationalStore" "Server=localhost;Database=MigrationOperationalStore;Trusted_Connection=True;TrustServerCertificate=True"
```

For a local smoke run, set these secrets or appsettings values:

```json
{
  "SqlOperationalQueueExecutor": {
    "Enabled": true,
    "WorkerId": "local-sql-operational-worker-01",
    "RunId": "00000000-0000-0000-0000-000000000000",
    "AutoStartRun": false,
    "BatchSize": 25,
    "LeaseSeconds": 300,
    "PollDelaySeconds": 5,
    "RunUntilIdleAndStop": true,
    "CompleteNoOpWorkItems": false
  }
}
```

Set `RunId` to a real SQL operational run ID before enabling execution.

## Run

```powershell
dotnet run --project .\src\Hosts\Migration.Hosts.SqlOperationalWorker\Migration.Hosts.SqlOperationalWorker.csproj --environment Development
```

## Important

This set makes the SQL work-item queue executable through QueueExecutor plumbing. It does not yet perform real connector writes per work item. The default executor is a safety adapter so the claim/complete/fail lifecycle can be validated first.
