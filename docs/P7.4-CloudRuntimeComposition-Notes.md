# P7.4 Cloud Runtime Composition Notes

P7.4 intentionally does not add a project-file mutation script and does not add inline package versions. It provides a conservative composition seam that wires the SQL operational runtime from P7.1-P7.3 into a host while leaving the concrete SQL provider registration to the cloud host.

## Why delegate-based SQL connection wiring?

The repo uses centralized package version management. Adding a hard-coded `Microsoft.Data.SqlClient` dependency inside a drop-in without verifying the current central package graph would increase breakage risk. This set therefore registers `ISqlOperationalConnectionFactory` through a delegate. The host can supply an opened `DbConnection` using whichever SQL client package/version is already approved centrally.

## Runtime stack after P7.4

1. P7.1: SQL operational tables and queue store.
2. P7.2: SQL queue runtime loop and run store.
3. P7.3: hosted worker service wrapper.
4. P7.4: DI composition, connection-factory seam, and readiness probe.

## Recommended next implementation step

P7.5 should create or wire the actual worker host entry point. That is the point where the repo-specific executable project should call `AddP7SqlOperationalRuntime(...)`, supply a SQL connection factory, and bind cloud configuration.
